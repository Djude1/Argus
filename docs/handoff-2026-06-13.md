## 三重授權鎖（理解這個才懂為何預設安全）

`validate_findings_with_kali` → `run_sqlmap` 只在**三者同時成立**才真的攻擊：
1. `settings.ARGUS_KALI_ENABLED=True`（只在 `docker-compose.attack.yml` override 開）
2. `scan_job.scan_mode == "active"`
3. `scan_job.active_testing_authorized == True`

→ 預設 `docker compose up`（公網生產）完全 inert，零影響。

## 唯一剩下的：實機端到端驗證

> ⚠ 實際打靶機是主動攻擊動作，Claude 的 harness 分類器會 gate sqlmap / docker exec 主動指令，
> 需由**人在場**從 Argus UI 觸發，或手動授權 Claude 執行。

### 步驟
1. 確認攻擊環境已起（使用者已做）：
   ```powershell
   docker compose -f docker-compose.yml -f docker-compose.attack.yml --profile attack up -d --build
   ```
2. （可選）無害管路驗證 — 證明 worker 特權路徑通、不打靶機：
   ```powershell
   docker exec argus-worker-1 docker exec argus-kali-1 sqlmap --version
   ```
3. 從 Argus UI（`localhost:8080`）對靶機 `https://htb.xn--gst.tw/` 建一個
   **scan_mode=active 且 active_testing_authorized=True** 的掃描。
4. 觀察 scan_log 應出現「Kali sqlmap 開始驗證…」與（若找到注入）「確認 N 項可利用漏洞」，
   報告中出現 `kali-sqlmap-sqli`（A03/CWE-89）critical Finding。

### 靶機資訊
- `https://htb.xn--gst.tw/`（CTF「宇宙錯誤知識管理局 / ERROR ARCHIVE」）
- 已知埋點：首頁 meta `flag{cekb-pii-003}`（PII 洩漏）、JS 暴露 `/api/debug/users|transactions|pay`
- sqlmap 需要帶 query 參數的 URL；若靶機注入點是 path/POST，可能要調整候選 URL 挑選邏輯
  （目前只挑含 `?...=` 的 GET URL）

## 關鍵地雷（別重犯）

- **`docker.sock` 不可加進基礎 `docker-compose.yml`** — 那份是公網生產部署用，會讓線上 worker 取得 host root。一律走 `docker-compose.attack.yml` override。
- **sqlmap 判定不可用 "injectable" 當特徵** — 找不到時它印 "do not appear to be injectable"，會假陽性。只用 "is vulnerable" / "injection point"。
- **commit 禁止 `git add .`** — 工作區常有 `.omc/`、`.codex/`、tmp、`ui-qa/` 等雜項，只 stage 明確路徑。
- 文件曾誤標 Phase 4「尚未開始」其實早已完成 — 以 `capstone-roadmap.md` 進度快照為準。
